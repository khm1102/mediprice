# MediPrice 기능 명세서 (MVP)

> **문서 목적** — 디자이너/프론트 개발자에게 다음 라운드(JSP + 카카오맵 + 정적 JS) 작업을 넘기기 위한 화면·플로우·데이터 명세.
> **기준일** — 2026-04-26 (P1 백엔드 완료 시점)
> **변경 이력** — 최초 작성

---

## 1. 서비스 개요

### 한 줄 정의
> **MediPrice는 "내 주변 병원의 비급여 진료비를 지도 위에서 비교"하는 공공데이터 기반 비교 검색 서비스다.**

### 목적과 배경
- 비급여 진료비(MRI, 도수치료, 임플란트 등)는 병원마다 가격이 제각각이지만 환자가 사전에 비교할 수단이 마땅치 않다.
- 심평원이 의료기관별 비급여 가격을 의무 공시하지만, 일반인이 직접 조회·비교하기는 어렵다.
- MediPrice는 심평원 공공 API를 수집·정제하고, **위치 기반 + 항목 기반**으로 가격을 빠르게 비교하도록 한다.

### 타겟 사용자 (MVP 한정)
- **수도권 중심 도시 거주자** — 통원 가능 거리(2~10km) 내 병원 비교가 가치 있는 사용자
- **계획된 진료를 받는 환자** — 응급이 아닌 MRI/도수치료/임플란트 등 사전 가격 비교가 의미 있는 시술군

### 핵심 가치 제안
1. **지도 한 화면에 가격 표기** — 마커에 병원명 + 가격을 함께 표시해 즉시 비교 가능
2. **항목 → 위치 → 결과 순의 짧은 동선** — 회원가입/로그인 없이 3 클릭 안에 결과 도달
3. **공공 데이터 기반의 신뢰성** — 심평원 신고가 그대로 노출, 임의 가공 없음

### MVP에서 의도적으로 제외한 것
| 기능 | 사유 | 후순위 |
|---|---|---|
| 회원가입/로그인 | 발표용 프로토타입 범위 축소 | P2 (보류) |
| 비회원 검색 횟수 제한 (Guest JWT) | 인증 전체가 P2로 미뤄짐 — 모든 `/api/**` 비인증 통과 | P2 |
| 즐겨찾기/리뷰/평점 | 회원 도메인 의존 | P2+ |
| AI 추천/유사 시술 추천 | 데이터 양·품질 부족 | P3+ |
| 결제/예약 연동 | 비교 검색 서비스 범위 밖 | 범위 외 |
| 미용 시술(보톡스/필러/지방흡입 등) 비교 | **심평원 의무 공시 대상 아님 → 데이터 부재** | 범위 외 |

## 2. MVP 범위

### 2.1 백엔드 — 완료 (P1)
| 항목 | 상태 |
|---|---|
| `GET /api/items` — 비급여 항목 그룹 (중분류 → 항목 리스트) | 완료 |
| `GET /api/hospitals?lat&lng&npayCd&radius` — 위치+항목 근거리 검색 (PostGIS, 거리 정렬, 최대 20건) | 완료 |

| 심평원 3개 API 일배치 (병원/항목/가격) | 완료 |
| PostGIS 위치 검색 프로시저 + GIST 인덱스 | 완료 |

## 3. 사용자 페르소나

### 페르소나 A — "MRI 가격 알아보러 온 직장인 김민수(34, 서울 강남구)"
- **상황** — 허리 통증으로 정형외과 권유받음. MRI 척추 검사가 필요한데 병원마다 가격 차이가 30~50만 원이라 들음.
- **목표** — 회사(역삼동) 근처 도보·지하철 10분 거리 병원의 MRI 척추 가격을 비교하고, 가장 합리적인 곳을 골라 예약 전화하기.
- **MediPrice에서의 동선**
  1. 메인 화면에서 "MRI 척추" 카드 클릭
  2. 위치를 "현재 위치"로 허용 → 강남 일대 병원 마커가 가격순으로 뜸
  3. 가장 저렴한 3곳 비교 → 상세 페이지에서 진료과목/주차/전화번호 확인 후 전화

### 페르소나 B — "임플란트 견적 받고 싶은 대학원생 이지은(28, 서울 관악구)"
- **상황** — 어금니 임플란트 1개 권유. 동네 치과는 비싼 것 같아 다른 곳도 알아보고 싶음.
- **목표** — 관악구·동작구 안에서 임플란트 신고가가 공개된 치과 후보 추리기. 너무 싼 곳은 부담스러워서 중간 가격대를 보고 싶음.
- **MediPrice에서의 동선**
  1. 메인에서 "치과 임플란트" 카드 클릭
  2. 위치는 지도 드래그로 관악구 신림동 일대 지정
  3. 결과 카드에서 가격 슬라이더로 중간 가격대만 필터(P1 범위 외, P2 후보) → 일단 가격순 정렬 보고 직접 판단

> **페르소나 의도적 비포함** — 미용시술 사용자(데이터 없음), 지방 거주자(8개 시도 미수집), 응급환자(서비스 성격에 맞지 않음). 6장에서 UX 대응 명시.

## 4. 핵심 사용자 플로우

```
[랜딩]
   ├─ (a) 추천 카드 클릭 (예: "MRI 척추")
   │       → 항목 코드(npayCd) 결정됨
   │       → 위치 권한 요청 → 결과 페이지로 이동
   │
   └─ (b) "전체 항목 보기" → 항목 선택 모달
           → 중분류 펼침 → 항목 선택 → 위치 권한 → 결과 페이지

[결과]
   지도 + 좌측 리스트 + 상단 필터 바
   ├─ 마커 클릭 → 리스트 항목 하이라이트 + 인포윈도우
   ├─ 리스트 카드 클릭 → 상세 페이지로 이동
   ├─ 반경 변경 (1/2/3/5km) → 재검색
   └─ 항목 변경 → 항목 선택 모달 → 재검색

[상세]
   병원 기본 정보 + 가격 목록 + 진료과목/장비/주차
   └─ 뒤로 가기 → 결과 페이지(상태 보존)
```

### 4.1 데이터 흐름 (각 단계별)

| 단계 | 호출 API | 빈 상태 처리 |
|---|---|---|
| 랜딩 진입 | (없음, 정적) | — |
| 추천 카드 클릭 | (없음, 카드의 `npayCd` 사용) | — |
| 항목 선택 모달 | `GET /api/items` | 응답 빈 배열이면 "항목 데이터 준비 중" 안내 |
| 결과 진입 | `GET /api/hospitals?lat&lng&npayCd&radius` | 빈 배열이면 6장 참고 (반경 확장 제안 + 미수집 시도 안내) |
| 마커/카드 클릭 | (없음, 응답 데이터 활용) | — |
| 상세 진입 | `GET /api/hospitals/{ykiho}` | 404 시 "병원 정보를 찾을 수 없습니다" 페이지 |

## 5. 화면 명세

### 5.1 랜딩 페이지 (`/`)

#### 목적
- 사용자가 1초 안에 "여기서 무엇을 하는지" 파악하고, 3 클릭 안에 결과로 진입.

#### 와이어프레임 (모바일 우선)

```
┌────────────────────────────────────────┐
│  MediPrice               [메뉴]        │ ← 헤더 (로고 + 햄버거)
├────────────────────────────────────────┤
│                                        │
│   내 주변 병원의                       │
│   비급여 진료비를                      │ ← Hero 카피
│   지도로 비교하세요                    │
│                                        │
│   [전체 항목 보기 →]                   │ ← 1순위 CTA
│                                        │
├────────────────────────────────────────┤
│  추천 진료 항목                        │
│  ┌──────┐ ┌──────┐ ┌──────┐          │
│  │ MRI  │ │ 도수 │ │ 임플 │          │
│  │ 척추 │ │ 치료 │ │ 란트 │          │ ← 7장 추천 카드
│  └──────┘ └──────┘ └──────┘          │
│  ┌──────┐ ┌──────┐ ┌──────┐          │
│  │ MRI  │ │ 라식 │ │ 갑상 │          │
│  │ 관절 │ │ 라섹 │ │ 선   │          │
│  └──────┘ └──────┘ └──────┘          │
├────────────────────────────────────────┤
│  데이터 출처: 건강보험심사평가원       │ ← 푸터
│  © MediPrice (학교 팀 프로젝트)        │
└────────────────────────────────────────┘
```

#### 화면 요소
| 요소 | 데이터 | 액션 |
|---|---|---|
| 헤더 로고 | 정적 | 클릭 시 `/`로 이동 |
| Hero 카피 | 정적 | — |
| "전체 항목 보기" 버튼 | 정적 | 항목 선택 모달 오픈 (`GET /api/items`) |
| 추천 카드 (4~6개) | 정적 (npayCd 하드코딩) | 클릭 시 위치 권한 요청 → `/hospitals?lat&lng&npayCd=...&radius=2` |
| 푸터 | 정적 | — |

#### 위치 권한 요청 흐름
1. 사용자가 카드 또는 항목 선택 → `navigator.geolocation.getCurrentPosition` 호출
2. 허용 → 좌표 획득 후 `/hospitals?lat={lat}&lng={lng}&npayCd={...}&radius=2`로 이동
3. 거부/실패 → 모달: "위치를 직접 선택할 수 있어요" + 지도 모달(서울 시청 기본 중심) → 핀 드롭

### 5.2 항목 선택 모달

#### 와이어프레임

```
┌─────────────────────────────────────┐
│  진료 항목 선택            [닫기 ×]│
├─────────────────────────────────────┤
│  [검색 입력  ____________]          │ ← 클라이언트 필터링
├─────────────────────────────────────┤
│  ▼ 영상진단 (MRI, CT 등)            │
│      • MRI 척추                     │
│      • MRI 관절                     │
│      • CT (조영제 사용)             │
│  ▼ 처치/시술                         │
│      • 도수치료                     │
│      • 체외충격파                   │
│  ▶ 치과                              │ ← 접힌 그룹
│  ▶ 안과                              │
│  ▶ 제증명수수료 [숨김 옵션]         │ ← 6장: 노이즈 그룹 분리
└─────────────────────────────────────┘
```

#### 데이터
- `GET /api/items` → `List<NonPayItemGroupDto>` (mdivCdNm으로 그룹핑된 코드/이름 리스트)
- 응답 구조:
  ```json
  [
    { "mdivCdNm": "영상진단", "items": [{ "npayCd": "...", "npayKorNm": "MRI 척추" }, ...] },
    ...
  ]
  ```

#### 액션
- 검색 입력 → 클라이언트 측 `npayKorNm` 필터링
- 그룹 헤더 클릭 → 펼침/접힘 토글
- 항목 클릭 → 모달 닫고 위치 권한 흐름 진입

### 5.3 검색 결과 페이지 (`/hospitals?lat&lng&npayCd&radius`)

#### 와이어프레임 (모바일)

```
┌────────────────────────────────────────┐
│  ← MRI 척추 · 반경 2km    [항목변경]   │ ← 상단 컨텍스트 바
├────────────────────────────────────────┤
│  [지도 영역 — 카카오맵]                │
│                                        │
│   ●(45만)  ●(52만)                     │ ← 가격 마커
│         ●(38만)  ←내위치               │
│      ●(60만)                           │
│                                        │
│  [반경: 1·2·3·5km]   [내위치 재중심]   │ ← 지도 컨트롤
├────────────────────────────────────────┤
│  검색 결과 12개 · 가격순               │ ← 정렬·결과 카운트
│  ┌──────────────────────────────────┐ │
│  │ 1위 △△의원        38만원        │ │
│  │ 서울 강남구 ...   1.2km          │ │
│  │ ☎ 02-XXX-XXXX                    │ │
│  └──────────────────────────────────┘ │
│  ┌──────────────────────────────────┐ │
│  │ 2위 ○○병원        45만원        │ │
│  │ ...                              │ │
│  └──────────────────────────────────┘ │
└────────────────────────────────────────┘
```

#### 화면 요소
| 영역 | 컴포넌트 | 데이터/액션 |
|---|---|---|
| 상단 컨텍스트 바 | 뒤로가기, 항목명, 반경, 항목 변경 버튼 | 항목 변경 → 모달 재오픈 |
| 지도 | 카카오맵 SDK | 마커 클릭 → 리스트 항목 하이라이트 + 인포윈도우 |
| 가격 마커 | 커스텀 오버레이 (가격 표시) | `curAmt`를 만원 단위로 포맷 (`38만`) |
| 반경 컨트롤 | 1/2/3/5 km 토글 | 변경 시 URL 쿼리 갱신 + 재요청 |
| 내 위치 재중심 | 버튼 | `getCurrentPosition` 재호출 |
| 결과 리스트 | 카드 리스트 | 카드 클릭 → `/hospitals/{ykiho}`로 이동 |

#### 데이터
- `GET /api/hospitals?lat=..&lng=..&npayCd=..&radius=..` → `List<HospitalSummaryDto>`
- 필드: `ykiho`, `yadmNm`, `addr`, `clCdNm`, `telNo`, `curAmt`, `distance`, `lat`, `lng`
- 정렬은 서버에서 distance ASC, 최대 20건

#### 빈 상태 / 에러 (6장 상세)
- 결과 0건 → "이 지역에는 신고된 가격 데이터가 없어요" + 반경 확장 제안 + 미수집 시도 안내
- API 5xx → 토스트 + 재시도 버튼

#### 화면 요소
| 영역 | 데이터 (`HospitalDetailDto`) | 비고 |

| 병원명/종류/주소 | `yadmNm`, `clCdNm`, `addr` | — |
| 전화번호 | `telNo` | `tel:` 링크 |
| 홈페이지 | `hospUrl` | null이면 숨김 |
| 의사 수 | `drTotCnt` | null이면 숨김 |
| 가격 목록 | `prices: List<PriceItem>` | 진입 시 검색했던 npayCd가 있으면 상단 강조 (URL 쿼리로 전달) |
| 진료과목 | `dgsbjtList` | 빈 배열이면 섹션 숨김 |
| 의료장비 | `medOftList` | 빈 배열이면 섹션 숨김 |
| 교통/주차 | `trnsprtInfo` | 모든 필드 null이면 섹션 숨김 |
| 특수진료 | `spclDiagList` | 빈 배열이면 섹션 숨김 |

#### 외부 API 부분 실패 처리
- 백엔드(`HospitalDetailService`)가 4개 API 중 일부 실패 시 빈 리스트로 fallback
- 프론트는 빈 배열/null인 섹션을 통째로 숨겨 "데이터 없음" 노이즈를 줄인다.

1. **랜딩 페이지 안내 문구** (작게, 푸터 위)

## 8. 에러 / 엣지 케이스

### 8.1 검색 결과 페이지

| 케이스 | 동작 |
|---|---|
| 결과 0건 + 수집 시도 내부 | "주변 2km 안에 신고된 가격이 없어요" + 반경 5km로 자동 확장 제안 버튼 |
| 결과 0건 + 미수집 시도 | "이 지역(예: 강원)은 아직 데이터 수집 중" + 수도권으로 이동 버튼 |
| `lat`/`lng` 파라미터 누락 | 백엔드 400 → 프론트는 메인으로 redirect + 토스트 "위치 정보가 필요해요" |
| `npayCd` 누락 | 동일하게 메인으로 + 항목 모달 자동 오픈 |
| `radius` 범위 외 (예: 음수, 100km 초과) | 백엔드 400 → 프론트는 기본값 2km로 fallback + 토스트 |
| API 5xx | 토스트 "일시적 오류" + 재시도 버튼 + 지도/리스트는 직전 상태 유지 |
| 위치 권한 거부 | 메인의 "지도에서 직접 선택" 흐름으로 fallback |
| 카카오맵 SDK 로드 실패 | 리스트 영역만 표시 + "지도를 불러오지 못했어요" 안내 |

### 8.2 병원 상세 페이지

| 케이스 | 동작 |
|---|---|
| `ykiho` 404 (`HospitalNotFoundException`) | "해당 병원 정보를 찾을 수 없어요" 페이지 + 검색 결과로 돌아가기 버튼 |
| 외부 API 4개 모두 실패 | DB 정보(병원 기본 + 가격)만 표시, 진료과목/장비/교통/특수진료 섹션 통째로 숨김 |
| 외부 API 일부 실패 | 성공한 섹션만 표시, 실패 섹션 숨김 (사용자에게 "일부 정보 누락" 표기는 하지 않음 — 노이즈) |
| 가격 0건 | "이 병원은 신고된 비급여 가격이 없어요" 안내 + 다른 병원 보러 가기 버튼 |
| 전화번호 null | "전화걸기" 버튼 비활성 |

### 8.3 공통

| 케이스 | 동작 |
|---|---|
| 네트워크 오프라인 | 토스트 "인터넷 연결을 확인해주세요" |
| `ApiResponse.success=false` | `error.message` 토스트 노출, `error.code`는 콘솔 로그 |
| `traceId` (X-Trace-Id 헤더) | 5xx 토스트 하단에 작게 표기 (디버깅용, 운영 정책에 따라 숨김 가능 — TODO E2 참고) |

## 9. 다음 라운드 후보 (P2+)

> **현재 라운드 = 프론트(JSP+카카오맵+JS) 작업**. 아래는 그 다음 라운드 후보. 우선순위 높은 순.

### 우선순위 1 — 회원/인증 흐름 (P2 보류 중)
- Guest JWT 발급 + 검색 횟수 제한 (3회) — 가입 유도 장치
- 회원가입/로그인 (이메일+BCrypt), JWT HttpOnly 쿠키
- 관련 문서: `docs/authentication.md`

### 우선순위 2 — 가격 필터/정렬 강화
- 가격 범위 슬라이더 (최저~최고)
- 정렬 옵션: 거리순/가격순/이름순
- 가격 통계 (평균/중위/최저) 배지

### 우선순위 3 — 즐겨찾기 + 비교 장바구니
- 회원 도메인 의존 → P2 회원가입 이후
- 병원 ★ 즐겨찾기, 2~3개 병원 가격 가로 비교 표

### 우선순위 4 — 8개 미수집 시도 데이터 추가
- 외부 API timeout 원인 분석 후 시도별 페이징 크기 축소/재시도 전략
- 운영계정 신청(`TODO N3`) 선행

### 우선순위 5 — 캐시 전략 개선
- 위치 격자 키 캐싱(`Math.round(lat*1000)/1000.0`) — `TODO E-P1-2`
- NonPayItem 메모리 캐시 — `TODO N6`

### 우선순위 6 — AI/추천
- 사용자 위치+검색 이력 기반 항목 추천
- 데이터 충분히 쌓인 후 (P3+)

### 우선순위 7 — 리뷰/평점
- 회원 도메인 + 모더레이션 정책 필요
- 의료법 검토 필요 → 신중
